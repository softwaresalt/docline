title: UTF-8 BOM handling in Markdown frontmatter parsing
updated_at: 2026-07-05T21:28:02.2900421-07:00

Fix UTF-8 BOM handling in Markdown frontmatter parsing. A leading byte-order mark (U+FEFF) prefixed to a `.md` file defeats `_parse_md_frontmatter` in `src/docline/process/output_contract.py`: the BOM sits before the opening `---`, so the YAML frontmatter is not recognized, the whole `---\ntitle: ...` block is treated as body content, and the H1→H2→H3 heading validator then rejects it — the document falls back to raw-body output with NO assembled docline frontmatter and NO chunk anchors. Observed at scale during the 2026-07-05 ingest of Microsoft Learn corpora (bi-shared-docs, fabric-docs, powerbi-docs, query-docs), where many files carry a UTF-8 BOM (e.g. analysis-services tutorial pages), degrading graphtor-consumable frontmatter fidelity. Fix: strip a leading BOM before frontmatter detection (and read text as utf-8-sig where appropriate) so BOM-prefixed files assemble the same frontmatter as non-BOM files.

<!-- BEGIN:dod -->
- A `.md` file whose bytes begin with the UTF-8 BOM (EF BB BF) plus YAML frontmatter is parsed correctly: frontmatter is stripped, the body starts at the first heading, and `assemble_markdown` emits the docline frontmatter contract (no raw-body fallback).
- Fix is applied at the ingestion parse boundary (`_parse_md_frontmatter` and/or the `.md/.txt` read path in `build_output_document_parts`); consider `utf-8-sig` decoding for text/markdown reads.
- Regression test: a BOM+frontmatter fixture ingests to a document with the expected `title`/`doc_type` in frontmatter (not in the body).
- Re-run one previously-affected corpus (e.g. bi-shared-docs analysis-services tutorials) and confirm the 'Failed to build frontmatter' warnings are gone.
- Quality gates pass (ruff, pyright, pytest, format).
<!-- END:dod -->

<!-- BEGIN:goals -->
- BOM-prefixed Markdown files produce the same assembled docline frontmatter + chunk anchors as identical non-BOM files.
- Eliminate the 'H2 heading ... appeared before any H1' fallback caused by an unstripped BOM.
- Preserve graphtor-consumable frontmatter fidelity across the Microsoft Learn corpora.
<!-- END:goals -->

<!-- BEGIN:implementation-notes -->
- Root cause: `_parse_md_frontmatter` (src/docline/process/output_contract.py) checks for a leading `---` but a `\ufeff` precedes it; also `read_text(encoding='utf-8')` keeps the BOM (use `utf-8-sig` to drop it on decode).
- Minimal fix: `text = text.lstrip('\ufeff')` (or decode via utf-8-sig) before the frontmatter delimiter check; verify the DocFx include/normalize passes still see clean text.
- Evidence + reproduction: docs/memory/2026-07-05/050-F-openapi-ingestion-memory.md and the 2026-07-05 ingest logs; sample files under C:/Source/Docs/bi-shared-docs/docs/analysis-services/tutorial-tabular-1400/.
