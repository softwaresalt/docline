title: Surface OpenAPI pagination (x-ms-pageable) semantics in operation docs
updated_at: 2026-07-06T11:24:58.4195382-07:00

Add a Pagination section to the OpenAPI operation renderer that surfaces the `x-ms-pageable` extension (present in 52 fabric-rest-api-specs files): the next-page field (`nextLinkName`, e.g. `continuationUri`) and the optional items field (`itemName`). This lets graphtor distinguish collection/list operations and encode page-traversal semantics. Sole grounded residual promoted from the D9AC2CD6 OpenAPI epic; the other residuals (versioning/monikers, LRO, security deep-render, azure-wide sweep) are deferred with evidence in docs/decisions/2026-07-06-openapi-deferred-residuals-triage.md.

<!-- BEGIN:dod -->
- `render_operation` emits a Pagination section when `x-ms-pageable` is a mapping; omits it entirely when absent (section-omission parity with existing blocks).
- `Items field` line omitted when `itemName` is absent; `Next-page field` always rendered from `nextLinkName`.
- CLI and MCP produce identical output (both render via `render_operation`).
- TDD: unit tests cover present / absent / itemName-missing cases; a fabric-corpus assertion confirms a known pageable operation renders the section.
- No regression to existing OpenAPI render tests.
- Gates green: ruff check, pyright src/, pytest, ruff format --check.
<!-- END:dod -->

<!-- BEGIN:goals -->
Emit a `## Pagination` section from `render_operation` when an operation carries `x-ms-pageable`, surfacing `nextLinkName` (next-page field) and `itemName` (items field, when present), so downstream graph consumers can identify pageable list operations and traverse pages.
<!-- END:goals -->
