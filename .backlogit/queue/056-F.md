        Console output verbosity + progress reporting for `docline fetch` and `docline process`. Today both commands print only a single JSON result line at the very end — during a long web crawl (e.g. the ~1148-page PostgreSQL /docs/current ingest, ~30-40 min) the operator sees NO progress and must monitor process memory / output file counts externally. Add a verbosity flag to control console output and show live progress.

        MODES (industry-standard; pick a convention): either an enum `--verbosity {silent,normal,verbose}` OR the conventional `-q/--quiet` + `-v/--verbose` pair (default = normal). Rough behavior:
        - silent/quiet: no console output (final result still returned via exit code; optionally still print the terminal JSON to stdout for scripting).
        - normal (default): concise progress — periodic percentage / page counts, plus the final JSON summary.
        - verbose: per-page (fetch) and per-file (process) lines with URL/path + running percentage.

        PROGRESS PERCENTAGE:
        - process: total is known up front (len of _ordered_staged_files), so % = files_processed / total_files — straightforward.
        - fetch (crawl): estimate is approximate because the frontier grows during the BFS. Options: pages_fetched / max_pages (simple, bounded), or pages_fetched / (visited + remaining_frontier) as a moving estimate. Note the crawl currently accumulates all results in memory and only returns at the end (fetch/crawl.py::crawl), so a progress hook/callback must be threaded through crawl() -> _fetch_url (elt/execute.py) -> execute_source_configs / execute_fetch.

        DESIGN CONSTRAINTS:
        - Preserve dual-interface parity: keep the library functions (execute_fetch, execute_source_configs, execute_process, crawl) returning structured results and NOT printing directly, so the MCP surface stays clean/testable. Wire console output only at the CLI layer via an optional progress-callback / observer param (or a stdlib logging level mapped from --verbosity). Do not print from the library.
        - TTY awareness: use carriage-return in-place updates for a live percentage only when stdout is a TTY; emit plain newline-terminated lines when piped/redirected (no control chars in captured output). Respect NO_COLOR / not-a-tty.
        - Progress cadence: throttle updates (e.g. every N pages/files or every ~1s) to avoid console spam on large corpora.
        - Keep the terminal JSON result contract unchanged in normal/verbose (scripts may parse it); in silent, consider suppressing it or gating behind a separate --json flag.

        SCOPE/TESTS: add --verbosity (or -q/-v) to the fetch and process subparsers (cli.py); add an optional progress-reporter callback to the fetch/process execution paths; unit-test the reporter (percentage math, throttling, TTY vs non-TTY formatting) and the CLI flag parsing. Refs: src/docline/cli.py (fetch/process parsers + dispatch), src/docline/app.py::execute_process (per-file loop, known total), src/docline/elt/execute.py::_fetch_url + execute_source_configs, src/docline/fetch/crawl.py::crawl (needs a progress hook). Relates to the 054-F web-fetch capability.

Add a `-q/--quiet` + `-v/--verbose` verbosity control and live progress reporting to `docline fetch` and `docline process`. Progress is emitted to stderr (TTY-aware carriage-return updates; newline lines when piped); the terminal JSON result on stdout is unchanged in every mode. Library functions stay print-free — an optional `progress` callback (default None) is threaded through crawl → _fetch_url → execute_elt_fetch/execute_source_configs/execute_fetch and execute_process. fetch progress is count-authoritative (pages_fetched; max_pages is a budget/ceiling, never forced to 100% on early completion); process progress is a global cumulative files_done/total across ALL completed jobs (monotonic, job identity in detail); completion is emitted via ProgressReporter.finish() and never fabricates 100%.
