Unit 2a. Add an optional keyword `progress` callback to crawl() (src/docline/fetch/crawl.py) and invoke it once per processed page inside the BFS loop. Default None = exact current behavior. Test-first. Files: fetch/crawl.py + tests/fetch/test_crawl_progress.py.

<!-- BEGIN:acceptance-criteria -->
- `crawl()` accepts optional keyword `progress: Callable[[int, int | None, str], None] | None = None`.
- Callback invoked once per processed URL with `(page_count, crawl_config.max_pages, current_url)`; counts monotonic non-decreasing and ≤ max_pages.
- METRIC SEMANTICS (N-crawl-count): `page_count` is budget-consumed/attempted pages — the crawl increments it for robots-denied, failed, and rejected-redirect URLs too (crawl.py:145-191), NOT staged pages. The authoritative staged-page count (results with bodies) is `staged_count` in `_fetch_url` (elt/execute.py:499-526), surfaced to the CLI for `ProgressReporter.finish()` as the completion figure.
- `max_pages` is a budget/ceiling: the crawl routinely ends early with `done < total` and `crawl` forges no synthetic 100%.
- `progress=None` leaves the returned list of CrawlResult unchanged (characterization test).
- Tests: callback fires for a robots-denied/failed page (metric is attempted, not staged); early frontier exhaustion → last event `done < total`, no forced 100%.
<!-- END:acceptance-criteria -->
