title: Build OCR memory calibration harness (operator-run) to determine limits
updated_at: 2026-06-30T19:18:36.4115426-07:00
---

Build a calibration harness (e.g. scripts/study/ocr_memory_calibration.py) the operator runs to OBSERVE docling OCR peak memory behavior, so the OCR page/memory caps and render-scale schedule are set from measurement rather than guessed. This replaces the earlier (mis)plan of hardcoding a 0.5/0.25 scale schedule.

What it measures: for a set of representative pages (varying physical size / bitmap dimensions — e.g. a normal text page, a large/high-DPI scan, an oversized diagram page), sweep (a) docling render scale (ocr_scale via 040.001-T) and (b) pages-per-group, running docling OCR in the batched worker subprocess and recording peak RSS (e.g. via psutil on the child process) and success/OOM. Emit a TSV of {page_class, scale, pages_per_group, peak_rss_mb, outcome} plus a recommendation: the max pages-per-group that stays within a memory budget (derive the budget from resource_probe.recommended_docling_max_mb / available_ram_gb), and the render-scale step schedule that brings an oversized single page back under budget. Write a decision artifact (docs/decisions/) with the observed values.

Agent-buildable + unit-testable in the 036-S harness pattern: the harness LOGIC (arg parsing, sweep orchestration seams, TSV emission, recommendation computation from a measurements table) is unit-tested by loading the script via importlib and feeding synthetic measurement rows (cf. tests/test_compare_merge_gap.py). The real measurement RUN is operator-executed on a docling-enabled host (docling is an optional extra, not installed in CI); guard the docling/psutil measurement path behind availability checks so import + logic tests pass without docling. Depends on 040.001-T (needs the ocr_scale knob to sweep scales).

Do NOT hardcode the resulting cap/schedule in this task — that is the follow-up ("apply calibrated OCR limits") after the operator runs this harness.
