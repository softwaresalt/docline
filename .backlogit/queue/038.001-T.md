Add OCR-aware bounded grouping to src/docline/process/page_range.py so the batched docling worker isolates OCR-flagged items under a tighter memory cap. Currently group_by_page_count(page_counts, max_pages=MAX_BATCHED_PAGES=40) bins purely by cumulative page count, ignoring whether a group will run OCR. OCR rendering (RapidOCR) loads page bitmaps and is the memory driver behind the 0xC0000005 bad_alloc crash, so OCR-free items can safely fill a 40-page group while OCR items must be bounded much tighter.

Implement a new grouping function (e.g. group_by_page_count_ocr_aware(page_counts, needs_ocr, *, max_pages=MAX_BATCHED_PAGES, ocr_max_pages=OCR_MAX_BATCHED_PAGES)) that:
- never places an OCR-flagged item in the same group as an OCR-free item (so a crash on the OCR group cannot drag down OCR-free docling work);
- bins OCR-flagged items under the tighter ocr_max_pages cap (calibrate to a conservative single-page-ish bound; document the GB rationale alongside the existing MAX_BATCHED_PAGES comment);
- bins OCR-free items under the existing max_pages cap, preserving today's behavior;
- preserves document order within and across groups so downstream splice-back stays aligned;
- raises ValueError on max_pages<1 or ocr_max_pages<1, matching the existing contract.

Add a module constant OCR_MAX_BATCHED_PAGES with a calibrated value + rationale comment. Keep the existing group_by_page_count for the OCR-free-only path / backward compatibility (or have it delegate). This task is the page_range.py change + its unit tests ONLY; wiring into pdf_triage.py / pdf_batch.py is task 2.
