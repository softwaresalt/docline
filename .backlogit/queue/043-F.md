title: Per-page boundary markers in stitched batch markdown (graphtor-docs consumer)
updated_at: 2026-07-03T18:28:28.1469275-07:00
---

Inject opt-in per-page boundary markers (HTML comments like `<!-- page N -->`) into the stitched batch markdown produced by `pdf_batch`, so downstream graph writers can recover page boundaries from the stitched output. This was previously gated on the existence of a downstream consumer; graphtor-docs is now installed as an MCP ingestion consumer, making it relevant. Per-page granularity already exists in `ChunkResult.chunk_pages`; the work threads that into a page-aware stitch path.

<!-- BEGIN:dod -->
- A page-aware stitch path prepends a `<!-- page N -->` marker before each page's markdown when enabled; the default-off path is byte-identical to current output.
- Page numbering is monotonic and source-relative, correctly skipping `page_overlap`-duplicated boundary pages (or overlap handling is explicitly documented/deferred with a deterministic counter).
- Heuristic / empty `chunk_pages` chunks degrade gracefully with documented behavior.
- Unit tests cover enabled vs disabled, multi-page-per-chunk, overlap dedup, and heuristic fallback.
- All four quality gates pass: ruff check ., pyright src/, pytest, ruff format --check .
<!-- END:dod -->

<!-- BEGIN:goals -->
- Emit `<!-- page N -->` markers at page boundaries in the stitched batch markdown, driven by the per-page data already captured in `ChunkResult.chunk_pages`.
- Keep the capability opt-in (default off) so existing stitched output and consumers are byte-identical unless markers are explicitly requested.
<!-- END:goals -->
