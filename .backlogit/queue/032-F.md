Recover the docling batched-worker throughput win that 033-S surrendered (it reverted use_batched_worker default to per-chunk for memory safety) without reintroducing the OOM that caused 86/86 docling fallback on the cosmos corpus. Groups three medium-priority follow-ups from 032-S/033-S into one perf-recovery shipment: bounded sub-batching, partial-crash resilience, and empirical batch-size calibration that feeds the sub-batch page cap.

<!-- BEGIN:dod -->
- Bounded sub-batching groups the manifest by cumulative page count, spawns a fresh worker per group (reclaims torch memory between groups), and cosmos re-validation shows bounded memory + docling actually running (not 100% fallback).
- Each chunk is gated on its own output envelope; a partial group crash recovers the group's successful chunks.
- Batch-size probe executed and results committed to docs/decisions/2026-06-14-docling-batch-size-probe.md; layout_batch_size constant chosen and wired into _read_pdf_docling_pages if a clear winner emerges (>=20% throughput without >=50% RSS regression).
<!-- END:dod -->

<!-- BEGIN:goals -->
- Restore batched-mode cold-start/model-load amortization while keeping peak RSS bounded.
- Make batched mode resilient to partial subprocess crashes (do not discard successfully-written chunk envelopes).
- Ground the sub-batch page cap (MAX_BATCHED_PAGES) in empirical probe data rather than guesswork.
<!-- END:goals -->
