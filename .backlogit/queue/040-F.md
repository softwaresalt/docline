Complete the OCR-OOM recovery arc from 038-F. Adaptive group-downsizing (038.003-T) recovers the ACCUMULATION case (many page bitmaps resident in one batched group) by retrying at a smaller pages-per-group cap (8->4->2->1). But a SINGLE page whose rasterized bitmap alone exceeds OCR memory cannot be split below 1 page, so it still crashes and falls back to heuristic (OCR content lost). This feature adds the final recovery step: when a single-page OCR group still OOMs at cap==1, retry that page at a REDUCED docling render scale (lower effective DPI) before conceding to heuristic. RapidOCR/onnxruntime peak memory scales ~quadratically with bitmap dimensions, so a 0.5x scale step is ~4x less peak memory.

Grounded surface: src/docline/readers/pdf.py::_read_pdf_docling_pages configures PdfPipelineOptions(images_scale=2.0) — the page-render scale the OCR engine consumes. A per-invocation ocr_scale override lowers OCR input resolution (and thus peak memory). src/docline/_tools/docling_worker.py reads per-chunk fields from the --batch manifest and calls _read_pdf_docling_pages(do_ocr=...); it gains an optional per-chunk ocr_scale. src/docline/process/batch_dispatch.py::dispatch_batched_groups_with_retry currently concedes a crashed single-item OCR group to heuristic; it gains a scale-step retry for that case.

Note: like the 038.003-T retry and the OCR_MAX_BATCHED_PAGES=8 cap, the CODE MECHANISM is validated in CI via fake-runner simulation; the empirical memory reduction is validated operator-side in production (docling is an optional extra, not installed in CI).
