Complete the OCR-OOM recovery arc from 038-F, with all OCR page/memory limits determined EMPIRICALLY (testing + observation), not hardcoded. 038.003-T recovers the accumulation case via group downsizing; a single page whose bitmap alone exceeds OCR memory still crashes. The final recovery step is retrying such a page at a reduced docling render scale (lower effective DPI) before conceding to heuristic.

IMPORTANT (operator clarification 2026-06-30): the 8->4->2->1 pages-per-group sequence and the 0.5/0.25 scale steps were ILLUSTRATIVE examples, NOT prescribed values. The shipped OCR_MAX_BATCHED_PAGES=8 and the cap//2 halving are PROVISIONAL. The real per-group page cap, the downsizing sequence, and the render-scale schedule must be derived from measured docling OCR peak memory as a function of page bitmap dimensions and render scale.

This shipment delivers the MEASUREMENT + the MECHANISM PLUMBING (agent-buildable, unit-testable): (1) an ocr_scale knob through the docling worker; (2) a calibration harness that measures OCR peak RSS + pass/fail across render-scale x pages-per-group for representative page sizes and emits a recommended limit schedule. The harness is operator-run (docling is an optional extra, not in CI). APPLYING the observed limits (memory-aware group cap replacing the provisional 8, and the scale-step retry schedule) is a follow-up shipment after the operator runs the harness — mirroring the 036-S harness -> 036.002-T operator run -> apply pattern.
