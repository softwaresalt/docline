Wire the OCR-aware grouping (task 038.001-T) into both batched docling dispatch paths and add OOM blast-radius regression coverage.

In src/docline/process/pdf_triage.py (batched-splice path, ~lines 554-593): replace group_by_page_count(page_counts) with the OCR-aware grouping, passing the parallel range_do_ocr list (already computed at line 548) as the needs_ocr argument. In src/docline/process/pdf_batch.py (_run_chunks_batched, ~lines 274-309): compute a parallel needs_ocr list (the per-chunk _chunk_needs_ocr(chunks[i]) values are already computed when building each group's manifest at line 284 — compute them once up-front and reuse) and feed them to the OCR-aware grouping. Result: an OCR-enabled group is isolated; when its worker subprocess hard-crashes (non-zero exit, e.g. the 0xC0000005 OOM), only the OCR-flagged ranges/chunks in that group fall back to heuristic, while OCR-free ranges in their own group still receive docling output.

Keep per-range/per-chunk envelope gating in the post-pass unchanged (it already isolates per-item usability). No behavior change for OCR-free corpora (all items OCR-free -> single needs_ocr=False partition -> identical grouping to today).

Add regression tests: simulate a runner whose batched invocation returns a non-zero exit ONLY for the manifest/group that contains an OCR-flagged item, and assert (a) OCR-flagged ranges fall back to heuristic, (b) OCR-free ranges in the other group still get docling markdown, (c) OCR and non-OCR items were dispatched in separate --batch manifests.
