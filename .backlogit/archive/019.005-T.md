<!-- BEGIN:description -->
Extend `src/docline/process/output_contract.py` to record per-page engine attribution in part frontmatter `docline:` namespace and a manifest-level `triage_stats` summary.

Per-part field: `docline.engine` ∈ {`heuristic`, `docling`}.
Manifest summary: `triage_stats: {pages_total: N, pages_docling: N, pages_heuristic: N, flagged_ranges: N}`.

Fields are ABSENT for non-triage runs (no schema break; downstream consumers tolerate absence).

Files:
- `src/docline/process/output_contract.py` (modified)
- `tests/process/test_output_contract_engine_attribution.py` (new or extension)

Depends on: U3 (orchestrator produces TriageResult).
<!-- END:description -->

<!-- BEGIN:implementation-notes -->
This is the highest-coupling change in the shipment. Plan hardening Invariant #3 codifies the merge-not-overwrite rule.

Follow the existing `WebFrontmatter` pattern in `src/docline/app.py:_build_markdown_with_frontmatter` — it already routes multiple keys into `docline:` via a pydantic validator. The new `engine` field rides the same merge path; do NOT introduce a parallel write path that assigns to `payload_dict['docline']` directly (that destroys the existing namespace contents).

When TriageResult is unavailable (non-triage runs), the existing emission path is unchanged — only triage runs add the new fields.
