# docling_worker per-page output contract

## Problem Frame

<!-- BEGIN:problem-frame -->
The `docline._tools.docling_worker` subprocess receives a (possibly multi-page) PDF and returns a single flat-markdown blob via `"\n\n".join(pages)`. The orchestrator (pdf_triage._splice_and_run_docling) currently places that blob on the FIRST page index of the flagged range and emits empty strings for all subsequent pages of the range — yet still marks every page in the range as `engine="docling"`. This is a fidelity/attribution mismatch that surprises downstream consumers (pdf_batch.py stitcher, future graph-DB writers, anyone expecting `TriageResult.pages[i]` to carry the actual content for page i). The root cause is information loss at the worker's output serialization boundary: `_read_pdf_docling_pages` already returns `list[str]` (per-page), but the worker collapses it. The contract refresh must restore page-level fidelity without breaking the existing subprocess invocation model (isolation, fallback-to-heuristic, OS-reclaim semantics).
<!-- END:problem-frame -->

## Options

<!-- BEGIN:options -->
**Option 1 — JSON envelope from worker (RECOMMENDED)**: Change `docling_worker.main()` to write `{"pages": ["...","..."], "page_count": N}` JSON to the output path instead of flat markdown. Update pdf_batch.py and pdf_triage.py consumers to parse the envelope and index by page. The worker already has the per-page list internally; this is a pure serialization change. Effort: ~2-4 hours. Risk: low — internal contract, no public API surface change.

**Option 2 — One subprocess per page**: Run `docling_worker` once for every page in a flagged range instead of once per range. Per-page output is trivially correct because each subprocess output IS one page. Effort: small code change. Risk: HIGH — catastrophic perf regression. Cosmos PA3+PA4 had 86 subprocess invocations × 24 pp avg = 2064 invocations under Option 2. With ~5-10s model-load overhead per subprocess, total wall-clock balloons from 247 min to potentially 14+ hours. Directly contradicts the 51332802 perf-tuning goal.

**Option 3 — Keep flat output, add docs + heuristic splitter helper**: Leave worker contract unchanged. Add a prominent doc note in pdf_triage.py and ship a `split_blob_by_heading(blob, page_count)` helper that re-splits the blob on H1 boundaries when downstream needs per-page. Effort: very small for docs; medium for splitter helper + tests. Risk: medium — heuristic splitting is brittle (many PDFs don't have one H1 per page; tables/figures span page boundaries; engine attribution remains misleading because pages[1..N-1] are still "" despite engine="docling" claim). Perpetuates the fidelity debt rather than retiring it.
<!-- END:options -->

## Chosen Direction

<!-- BEGIN:chosen-direction -->
**Option 1 — JSON envelope from worker.** Decisive reasons: (a) the per-page data already exists inside the worker process; it is being discarded at the `"\n\n".join` boundary. Restoring it is essentially free. (b) Option 2 would impose a 10-30× perf regression that directly conflicts with the paired 51332802 perf-tuning work. (c) Option 3 papers over the fidelity gap with a brittle splitter and leaves the engine attribution misleading. (d) The JSON envelope is the structural prerequisite for the 51332802 multi-chunk batched worker mode (one invocation processes multiple chunks, output keyed by chunk id), so Option 1 pays double dividends. Backward-compat note: the worker output is an internal subprocess artifact — only pdf_batch.py and pdf_triage.py read it. No public API contract is broken. The worker exit-code contract (0/2/3/4/5/6) is preserved unchanged.
<!-- END:chosen-direction -->

## Open Questions

<!-- BEGIN:open-questions -->
1. Should the envelope file extension stay `.md` (current) or move to `.json`? Recommendation: keep `.md` extension for orchestrator-side path conventions but write JSON content. Document the actual format in worker docstring. Alternative: rename to `.json` and update all chunk_out / splice_md path builders. Cleaner but ripples to all tests that reference the file pattern.

2. Should the multi-chunk batched worker mode (51332802 Strategy A) extend the envelope to `{"chunks": {"<id>": {"pages": [...]}}}` shape, or use one envelope per chunk written to separate output paths? Recommendation: per-chunk separate paths in the first implementation (preserves orchestrator's existing path-routing assumptions). Multi-chunk-per-file is a later optimization if the per-chunk file IO becomes hot.

3. Should `_stitch_chunk_markdown` in pdf_batch.py become per-page-aware so the stitched markdown carries page markers (e.g. HTML comments `<!-- page 5 -->`)? Out of scope for this shipment — file as low-priority follow-up if downstream graph writers need them.
<!-- END:open-questions -->

## Notes

<!-- BEGIN:notes -->
**Codebase grounding evidence**:

1. `src/docline/_tools/docling_worker.py:77-85` — `pages = _read_pdf_docling_pages(input_path)` returns `list[str]` (one entry per page). Then line 85: `markdown = "\n\n".join(pages)` collapses it. This is the exact loss point.

2. `src/docline/process/pdf_triage.py:470-483` — splice-back loop reads `splice_md.read_text()` as one blob, assigns to `final_pages[start]`, sets `final_pages[start+1..end]` to `""`, sets `engine_per_page[start..end]` to `"docling"`. The fidelity/attribution mismatch is here.

3. `src/docline/process/pdf_batch.py:197-204` — `_process_one_chunk` reads `output_path.read_text()` as one blob and stitches via `_stitch_chunk_markdown()`. The stitcher does H1-boundary deduplication, NOT per-page indexing. Stitcher output is stitched_markdown (a single string), not per-page. So pdf_batch consumers don't currently NEED per-page; they need stitched. Migration plan: pdf_batch's stitcher continues to consume `result["pages"]` joined locally, OR a new `stitched_markdown` field is added to the envelope. Cleanest: include both `pages: list[str]` and `text: str` in the envelope so pdf_batch.py can use either.

**Envelope schema (proposed)**:
```json
{
  "pages": ["page 1 markdown", "page 2 markdown", "..."],
  "page_count": N,
  "text": "joined markdown for legacy convenience"
}
```

The `text` field is the existing `"\n\n".join(pages)` value, kept for the pdf_batch.py stitcher to consume without re-joining. The `pages` field is what pdf_triage.py needs for per-page splice-back. The `page_count` field is a redundancy check.

**Migration safety**:
- Old fallback path (subprocess exit != 0 → heuristic) is unchanged.
- A single test fixture per consumer (pdf_batch and pdf_triage) updates from "read text file" to "read JSON file".
- Worker exit codes unchanged; on failure to serialize JSON, exit code 6 (unexpected error).
<!-- END:notes -->
