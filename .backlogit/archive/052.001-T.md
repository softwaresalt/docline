title: Strip UTF-8 BOM in Markdown frontmatter parsing + text read
updated_at: 2026-07-05T21:37:45.9107017-07:00

Strip a leading UTF-8 BOM (U+FEFF) so BOM-prefixed Markdown files parse frontmatter correctly. Fix `_parse_md_frontmatter` (src/docline/process/output_contract.py) to drop a leading BOM before the `---` fence check, and read `.md/.txt` staged files with `utf-8-sig` in `build_output_document_parts` so the BOM never reaches the docfx-include, normalize, or frontmatter passes. Add regression coverage.

<!-- BEGIN:acceptance-criteria -->
- `_parse_md_frontmatter('\ufeff---\ntitle: X\n---\n# H1')` returns the parsed mapping + a body starting at `# H1` (not None).
- A BOM-prefixed file with no frontmatter returns `(None, body)` with the BOM stripped from the body.
- End-to-end: a staged BOM+frontmatter `.md` produces output whose YAML frontmatter carries the title/doc_type (not the raw `---` block in the body); no 'appeared before any H1' fallback.
- Non-BOM files are unchanged (byte-for-byte for a representative fixture).
- ruff/pyright/pytest/format all pass.
<!-- END:acceptance-criteria -->

<!-- BEGIN:implementation-notes -->
- `_parse_md_frontmatter`: after the empty-text guard, `if text.startswith('\ufeff'): text = text[1:]`.
- `build_output_document_parts` .md/.txt branch: `file_path.read_text(encoding='utf-8-sig', errors='replace')`.
- Keep HTML/else branches as-is (out of scope). Verify docfx include/normalize passes still behave.
