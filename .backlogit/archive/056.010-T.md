Unit 1. New module (src/docline/progress.py or src/docline/observability/progress.py) with a Verbosity enum, a small ProgressEvent, and a pure TTY-aware ProgressReporter that renders/throttles progress to a stream (default sys.stderr). Test-first. Files: the new module + tests/.../test_progress.py.

<!-- BEGIN:acceptance-criteria -->
- `Verbosity` enum (SILENT/NORMAL/VERBOSE) defined; caller resolves default NORMAL.
- `ProgressReporter.__call__(done: int, total: int | None, detail: str)` builds a `ProgressEvent` internally (P2-1). `total: int | None` supports count-only reporting (N6).
- Percentage clamps `done/total` to [0,100]; `total is None` → count-only output (no ETA/percent). `total is None` is the ONLY count-only case — a known but incomplete total keeps its true `done/total` ratio.
- Stream `isatty()` true → carriage-return in-place update; false → newline-terminated line with no control chars.
- Rendering differs by mode (RC-5): NORMAL = throttled concise single percentage/count line; VERBOSE = one line per item with detail (URL/path) + running percentage, NOT throttled (never dropped); SILENT = nothing.
- `finish(detail=None)` emits the final line unconditionally (bypassing throttle) but NEVER forces 100% (RC-4/N4): for a KNOWN total it renders `done/total` (100% only when `done == total`, else the true ratio e.g. 40% for 20/50); a bare count ONLY when `total is None`. Fetch delivers completion via a `total=None` staged-count event (count-only); process has a known total and reaches 100% at `done == total`.
- Unit tests: percentage math + clamp + None-total count-only; NORMAL throttling coalesces; VERBOSE one detailed line per item; TTY vs non-TTY; SILENT no-op; `finish()` renders `done/total` for a known total (100% only at `done == total`, true ratio otherwise) and a bare count only when `total is None`.
<!-- END:acceptance-criteria -->
