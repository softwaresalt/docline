title: Adaptive OCR-group downsizing retry on OOM
updated_at: 2026-06-29T21:31:34.3191341-07:00
---

Add adaptive OCR-group downsizing retry on OOM to the batched docling dispatch. Today, when an OCR group's worker subprocess hard-crashes (non-zero exit, e.g. 0xC0000005 OOM), the crashed ranges/chunks fall back to heuristic with NO retry. This task makes recovery automatic: on a crash of a group containing OCR, re-split that group's items at half the cap and retry recursively (8->4->2->1) before conceding to heuristic. Successful smaller retries write per-item envelopes, which the existing post-pass already splices back as docling, so the recovery is transparent.

Implementation: new module src/docline/process/batch_dispatch.py exposing dispatch_batched_groups_with_retry(groups, *, inputs, outputs, do_ocr, page_counts, runner, manifest_dir, ocr_max_pages=OCR_MAX_BATCHED_PAGES) -> list[int] (per-item returncode). Wire it into both pdf_triage.py (batched-splice) and pdf_batch.py (_run_chunks_batched), replacing their inline per-group dispatch loops. The retry halves the cap each level; a single OCR item (len==1) or cap==1 that still crashes is left without an envelope so the caller's post-pass falls back to heuristic. Terminates because the cap halves each level regardless of regrouping progress.

Out of scope (stash follow-up): per-page OCR DPI/bitmap downscale for the single-page-too-large case that group-halving cannot fix.
