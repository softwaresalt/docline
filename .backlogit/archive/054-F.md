Wire the already-built fetch building blocks (fetch/crawl.py, http.py, sitemap.py, staging.py, url_policy/url_canonical) into the currently-stubbed `execute_fetch` (app.py) so docline can fetch/crawl online HTML end-to-end, then verify on PostgreSQL /docs/current. Today `execute_fetch` returns "Fetch execution is not implemented"; the processing path for staged `.html` already works (html_extract/html_normalize, WebFrontmatter, crawl-manifest). Probed 2026-07-06: postgresql.org robots.txt allows /docs/current/ and advertises a sitemap; docs are static HTML with relative .html links (no JS render needed). See docs/decisions/2026-07-06-postgresql-online-docs-fetch-capability.md.

<!-- BEGIN:dod -->
- `execute_fetch` runs a bounded crawl and persists each fetched page to `{staging}/…/files/<slug>.html` with a complete `metadata.json` (StagingJob) + crawl-manifest, then returns `FetchResult(staged_path, success=True)`.
- `FetchRequest.depth` maps to `CrawlConfig.max_depth`; a bounded default `max_pages` prevents runaway crawls.
- CLI `fetch` and MCP `fetch` produce identical staged output (parity).
- A subsequent `execute_process` run turns the staged PostgreSQL HTML into schema-valid Markdown with correct WebFrontmatter, canonical URL, tables, `<pre>` code fences, admonitions, and cross-doc graph edges.
- Deterministic tests use recorded fixtures (no live network in CI); URL policy / robots / SSRF guards remain enforced.
- All quality gates pass: `ruff check .`, `pyright src/`, `pytest`, `ruff format --check .`.
