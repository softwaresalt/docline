Add content-sniff detection for OpenAPI/Swagger specs and route them to a dedicated pipeline. A source is OpenAPI when the parsed root has an `openapi:` (3.x) or `swagger:` (2.0) key. Extension alone is insufficient — `.json`/`.yaml` are overloaded and config/`docfx.json` sidecars are already skipped by the process pass. Add `SourceKind.OPENAPI` and wire routing so OpenAPI specs bypass the extension-driven FILE path and target the new reader.

<!-- BEGIN:acceptance-criteria -->
- An OpenAPI 3.x document (root has `openapi:` key) classifies as `SourceKind.OPENAPI`.
- A plain config `.json` (no `openapi:`/`swagger:` root key) does NOT classify as OPENAPI.
- A Swagger 2.0 document (root has `swagger:` key) is detected (rendering deferred to a later task, but detection recognizes it).
- Existing FILE-path classification for `.docx/.pdf/.html/.md/.txt` is unchanged.
- Unit tests cover positive (OpenAPI 3.x), negative (config JSON), and boundary (empty/malformed) cases; tests written first and observed to fail (TDD).
<!-- END:acceptance-criteria -->

<!-- BEGIN:implementation-notes -->
- `router.classify_source` currently maps to `SourceKind` (URL/TRANSCRIPT/FILE); live dispatch is extension-driven in `app.py` via `_SUPPORTED_EXTENSIONS`.
- Add `SourceKind.OPENAPI` to the enum; sniff must parse minimally (peek root keys) without fully loading/rendering — keep it cheap.
- Content-sniff must handle both JSON and YAML roots (PyYAML already available).
- Route OPENAPI to `readers/openapi.py` (created in T2/T3); T1 only wires the classification + dispatch seam.
- Reference: spike design table 'Detection' + 'Source kind' rows (docs/decisions/2026-07-04-openapi-swagger-source-type-spike.md).
