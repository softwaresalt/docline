## Core Principles

### I. Safety-First Python (NON-NEGOTIABLE)

All production code MUST be written in Python (3.12).
type hints required on all public interfaces. ruff check fails on any warning; zero-tolerance for lint errors.
All exceptions must be typed; use custom exception hierarchy.

### II. Test-First Development (NON-NEGOTIABLE)

Every feature or chore MUST have tests written before implementation code. The test
directory structure (Tests in tests/ directory mirroring src/ structure) MUST be maintained. All tests MUST
pass via `pytest` before any code is merged.

### III. Workspace Isolation and Security Boundaries

All file-system operations MUST resolve within the configured workspace root.
Path traversal attempts MUST be rejected.

### IV. CLI Workspace Containment (NON-NEGOTIABLE)

Agents MUST NOT create, modify, or delete any file or directory outside the
current working directory tree.

### V. Destructive Command Approval (NON-NEGOTIABLE)

All destructive terminal commands MUST require operator approval before
execution, regardless of permissive mode flags. Destructive commands include:
file/directory deletion, force-overwrite, system config changes, VCS history
alteration, data drops, system-level package changes, and untrusted code execution.

### VI. Safety Modes for Risky Work

When work has elevated blast radius, agents MUST enter an explicit safety mode before proceeding:

* **careful** — enumerate risks and require approval before destructive or high-impact actions
* **freeze-scope** — constrain edits to a declared directory or subsystem boundary
* **investigate-first** — gather evidence before proposing or applying fixes

When the `strict-safety` capability pack is installed, agents MUST also follow
`.github/instructions/strict-safety.instructions.md`: express risky work as
`ProposedAction`, `ActionRisk`, and `ActionResult` rather than leaving the
approval path or rollback state implicit.

### Capability Overlay — strict-safety

When the `strict-safety` capability pack is installed, agents MUST also follow
`.github/instructions/strict-safety.instructions.md` and treat risky work as an
explicit review surface:

* classify risky steps as `ProposedAction` entries before execution
* assign the most conservative `ActionRisk` and route approval accordingly
* keep `ActionResult` visible as work moves from planning to review and closure
* require explicit approval for destructive actions and prefer approval for high-blast-radius actions
* carry risky action records into verification and closure when rollout safety depends on them

## Session Start Protocol

1. Ping intercom to establish heartbeat visibility; if it fails, declare `INTERCOM_DEGRADED`.
2. Verify engram workspace binding before discovery; use `unified_search` for pre-planning; if binding or search is unavailable, declare `ENGRAM_DEGRADED`.
3. Pull the backlogit queue before planning and honor dependency edges when selecting work.
4. Classify any risky or destructive step as a `ProposedAction` with `ActionRisk` before execution.
5. Broadcast phase transitions so remote operators can follow the session lifecycle.

### Capability Overlay Interaction Rules

When multiple capability packs are enabled simultaneously, follow these
precedence rules to resolve conflicts:

| Combination | Resolution |
|---|---|
| agent-intercom + backlogit | Query backlogit first for task state; format the result using intercom broadcast rules for remote operator choice |
| agent-engram + any other overlay | Engram is a discovery method; other overlays are independent. Use engram for search, other overlays for their domain |
| strict-safety + any overlay | Classify risky actions via strict-safety first; then route approval through the appropriate overlay (intercom if available, else local) |
| strict-safety + concurrency | Log lock conflicts as `ActionResult: blocked` when strict-safety is enabled |
| backlogit + strict-safety | Persist strict-safety decisions via backlogit checkpoints when checkpoint operations are supported |
| Multiple overlays unavailable | Fall back to core protocols (constitution + workflow-policies). Warn that overlay visibility is degraded |

General rules:

* Each overlay controls its own domain — do not attempt to merge protocols
* When in doubt about precedence, choose the more conservative behavior
* If an overlay's tool surface becomes unavailable mid-session, warn the
  operator and continue with the core protocol only
* Overlay interaction is additive: each adds behavior, none removes another

## Development Workflow

1. **Harness before code**: Every feature or chore MUST have a compiling but failing
   test harness before implementation begins.
2. **Backlog-driven planning**: All task tracking uses the configured backlog workspace and registry-backed operations; do not invent parallel task stores.
3. **Branch per release unit**: Each feature or chore on a dedicated branch.
4. **Commit discipline**: Conventional commits format (`feat:`, `fix:`, `docs:`, `test:`).
5. **No dead code**: Placeholder modules replaced or removed before completion.
6. **Closure before forgetfulness**: Runtime verification and operational closure happen before work is considered fully absorbed.

### Task Granularity (NON-NEGOTIABLE)

* **2-Hour Rule**: Every task scoped to roughly 2 hours of human effort
* **Width Isolation**: Each task targets a single skill domain
* **Atomic Milestone**: Every task produces a verifiable state change

### Stop Conditions

| Counter                    | Limit | Action                                             |
|----------------------------|-------|----------------------------------------------------|
| Tasks attempted in session | 20    | Halt, checkpoint, exit                             |
| Consecutive failures       | 3     | Halt, escalate or request help                     |
| Review-fix cycles per task | 3     | Accept remaining as backlog, commit, move on       |
| Fix-CI cycles              | 5     | Halt, leave PR for manual intervention             |

steps causes knowledge loss across sessions:

1. Write session memory to `docs/memory/` — include task IDs completed, files modified, decisions, failed approaches, and next steps
2. Capture compound learnings via the compound skill when hard-won solutions were discovered
3. Refresh or mark stale compound learnings when shipped work invalidates existing guidance in `docs/compound/`

5. Run quality gates (if code changed)
6. Update work item status via backlog tools
7. Push to remote (MANDATORY)
8. Provide handoff context for next session
